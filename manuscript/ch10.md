# Functional-Light JavaScript
# Глава 10: Функциональное асинхронное программирование

На этом этапе книги у вас уже есть все исходные концепции для основ ФП, которые я называю «Functional-Light Programming». В этой главе мы применим эти концепции в другом контексте, но принципиально новых идей здесь почти не будет.

До сих пор почти всё, что мы делали, было синхронным: мы вызывали функции с немедленными входными данными и немедленно получали выходные значения. Много задач можно решить таким образом, но этого совершенно недостаточно для полноценного современного JS-приложения. Чтобы быть по-настоящему готовым к ФП в реальном мире JS, нужно понимать асинхронное ФП.

Наша цель в этой главе — расширить наш способ думать об управлении значениями в ФП, распределив подобные операции во времени. Мы увидим, что Observable (и Promise!) — один из лучших способов сделать это.

## Время как состояние

Самое сложное состояние во всём вашем приложении — это время. То есть управлять состоянием значительно проще, когда переход из одного состояния в другое происходит немедленно и находится под вашим контролем. Когда состояние приложения меняется неявно в ответ на события, разбросанные во времени, управление становится экспоненциально сложнее.

Каждая часть того, как мы излагали ФП в этом тексте, была направлена на то, чтобы сделать код легче читаемым через повышение его надёжности и предсказуемости. Когда вы вводите асинхронность в свою программу, эти усилия получают серьёзный удар.

Но давайте будем точнее: беспокойство вызывает не сам факт того, что некоторые операции не завершаются синхронно; инициировать асинхронное поведение легко. Это координация ответов на эти действия, каждое из которых потенциально может изменить состояние приложения, требует столько дополнительных усилий. 

Итак, лучше ли, чтобы автор взял на себя эти усилия, или же оставить читателю кода разбираться, каким будет состояние программы, если A завершится до B или наоборот? Это риторический вопрос, но с вполне конкретным ответом с моей точки зрения: чтобы хоть как-то надеяться сделать такой сложный код более читаемым, автор должен проявлять значительно больше заботы, чем обычно.

### Сокращение времени

Один из важнейших результатов паттернов асинхронного программирования — упрощение управления изменением состояния путём абстрагирования времени из сферы нашей заботы. Для иллюстрации давайте сначала рассмотрим сценарий, в котором существует состояние гонки (race condition, иначе говоря, временнáя сложность), которое приходится обрабатывать вручную:

```js
var customerId = 42;
var customer;

lookupCustomer( customerId, function onCustomer(customerRecord){
    var orders = customer ? customer.orders : null;
    customer = customerRecord;
    if (orders) {
        customer.orders = orders;
    }
} );

lookupOrders( customerId, function onOrders(customerOrders){
    if (!customer) {
        customer = {};
    }
    customer.orders = customerOrders;
} );
```

Колбэки `onCustomer(..)` и `onOrders(..)` находятся в бинарном состоянии гонки. Предполагая, что оба выполнятся, любой из них может выполниться первым, и предсказать это невозможно.

Если бы мы могли вставить вызов `lookupOrders(..)` внутрь `onCustomer(..)`, мы были бы уверены, что `onOrders(..)` выполняется после `onCustomer(..)`. Но мы не можем этого сделать, поскольку нам нужно, чтобы оба запроса выполнялись конкурентно.

Поэтому для нормализации этой временнóй сложности состояния мы используем пару проверок `if` в соответствующих колбэках вместе с внешней лексически замкнутой переменной `customer`. Когда выполняется каждый колбэк, он проверяет состояние `customer` и тем самым определяет свой относительный порядок: если для колбэка `customer` не установлен, он выполняется первым, иначе — вторым.

Этот код работает, но с точки зрения читаемости далёк от идеала. Временнáя сложность затрудняет чтение. Давайте вместо этого используем Promise в JS, чтобы вынести время за скобки:

```js
var customerId = 42;

var customerPromise = lookupCustomer( customerId );
var ordersPromise = lookupOrders( customerId );

customerPromise.then( function onCustomer(customer){
    ordersPromise.then( function onOrders(orders){
        customer.orders = orders;
    } );
} );
```

Колбэк `onOrders(..)` теперь находится внутри колбэка `onCustomer(..)`, поэтому их относительный порядок гарантирован. Конкурентность запросов обеспечивается отдельными вызовами `lookupCustomer(..)` и `lookupOrders(..)` до указания обработки через `then(..)`.

Может быть не очевидно, но в этом фрагменте всё равно существовало бы состояние гонки, если бы не то, как определено поведение Promise. Если запрос `orders` завершается до того, как `ordersPromise.then(..)` вызывается для предоставления колбэка `onOrders(..)`, *нечто* должно быть достаточно умным, чтобы хранить этот список `orders` до тех пор, пока не будет вызван `onOrders(..)`. По факту, то же самое справедливо для `customer` — он должен быть доступен до того, как `onCustomer(..)` указан для его получения.

Это *нечто* — та же самая логика временнóй сложности, о которой мы говорили с предыдущим фрагментом. Но нам не нужно беспокоиться ни о чём из этой сложности — ни при написании кода, ни, что ещё важнее, при его чтении — потому что промисы берут на себя эту временнýю нормализацию.

Promise представляет единственное (будущее) значение независимым от времени образом. Более того, извлечение значения из промиса является асинхронной формой синхронного присваивания (через `=`) немедленного значения. Иными словами, промис растягивает операцию присваивания `=` во времени, но надёжным (независимым от времени) способом.

Теперь мы рассмотрим, как аналогично можно асинхронно растянуть во времени различные синхронные ФП-операции из предыдущих частей книги.

## Жадно vs. Лениво

«Жадный» и «ленивый» в мире информатики — не комплименты и не оскорбления, а лишь способы описать, завершится ли операция немедленно или будет продвигаться со временем.

ФП-операции, рассматриваемые в этом тексте, можно охарактеризовать как жадные, поскольку они работают синхронно (прямо сейчас) на дискретном немедленном значении или списке/структуре значений.

Вспомним:

```js
var a = [1,2,3]

var b = a.map( v => v * 2 );

b;          // [2,4,6]
```

Это отображение из `a` в `b` является жадным, потому что оно работает со всеми значениями в массиве `a` в данный момент и создаёт новый массив `b`. Если вы позднее измените `a` (например, добавив новое значение в конец), содержимое `b` не изменится. Это жадное ФП.

Но как выглядело бы ленивое ФП? Рассмотрим нечто вроде этого:

```js
var a = [];

var b = mapLazy( a, v => v * 2 );

a.push( 1 );

a[0];       // 1
b[0];       // 2

a.push( 2 );

a[1];       // 2
b[1];       // 4
```

`mapLazy(..)`, который мы здесь воображаем, по сути «слушает» массив `a`, и каждый раз, когда в его конец добавляется новое значение (через `push(..)`), он запускает функцию отображения `v => v * 2` и помещает преобразованное значение в массив `b`.

**Примечание:** Реализация `mapLazy(..)` не показана, поскольку это вымышленная иллюстрация, а не реальная операция. Для создания такого ленивого сопряжения между `a` и `b` нам понадобится нечто более умное, чем обычные массивы.

Представьте преимущества возможности сопрячь `a` и `b` таким образом, что каждый раз (даже асинхронно!), когда вы помещаете значение в `a`, оно преобразуется и проецируется в `b`. Это та же декларативная ФП-мощь операции `map(..)`, но теперь она может растягиваться во времени; вам не нужно знать все значения `a` *прямо сейчас*, чтобы настроить отображение из `a` в `b`.

## Реактивное ФП

Чтобы понять, как мы могли бы создать и использовать ленивое отображение между двумя наборами значений, нам нужно немного абстрагировать наше представление о списке (массиве). Давайте представим более умный вид массива — не просто хранящий значения, а лениво получающий и реагирующий («реагирующий», aka «реактивный») на значения. Рассмотрим:

```js
var a = new LazyArray();

var b = a.map( function double(v){
    return v * 2;
} );

setInterval( function everySecond(){
    a.push( Math.random() );
}, 1000 );
```

Пока этот фрагмент ничем не отличается от обычного массива. Единственное необычное то, что мы привыкли к тому, что `map(..)` работает жадно и немедленно создаёт массив `b` со всеми текущими отображёнными значениями из `a`. Таймер, помещающий случайные значения в `a`, странен, поскольку все эти значения приходят *после* вызова `map(..)`.

Но этот вымышленный `LazyArray` другой; он предполагает, что значения будут поступать по одному, со временем; просто помещайте значения через `push(..)` когда захотите. `b` будет ленивым отображением тех значений, которые в конечном итоге окажутся в `a`.

Кроме того, нам не нужно хранить значения в `a` или `b` после их обработки; этот особый вид массива хранит значение только до тех пор, пока оно нужно. Поэтому такие массивы не увеличиваются неограниченно в памяти со временем — важная характеристика ленивых структур данных и операций. По факту, это скорее не массив, а буфер.

Обычный массив является жадным — он хранит все свои значения прямо сейчас. «Ленивый массив» — это массив, значения которого будут поступать со временем.

Поскольку мы не всегда будем знать, когда в `a` поступило новое значение, нам также нужна возможность слушать `b`, чтобы получать уведомления о появлении новых значений. Мы могли бы представить слушатель вроде этого:

```js
b.listen( function onValue(v){
    console.log( v );
} );
```

`b` *реактивен* в том смысле, что он настроен *реагировать* на значения по мере их поступления в `a`. Существует ФП-операция `map(..)`, описывающая, как каждое значение передаётся из источника `a` в цель `b`. Каждая дискретная операция отображения — это именно то, как мы моделировали операции с единственным значением в обычном синхронном ФП, но теперь мы растягиваем появление значений во времени.

**Примечание:** Термин, чаще всего применяемый к этим концепциям, — Функциональное Реактивное Программирование (FRP). Я намеренно избегаю этого термина, поскольку существуют споры о том, действительно ли ФП + Реактивность составляют FRP. Мы не будем полностью погружаться во все подтексты FRP здесь, поэтому я продолжу называть это реактивным ФП. Альтернативно, вы могли бы назвать это событийно-управляемым ФП, если это менее запутанно.

Мы можем рассматривать `a` как производящий значения, а `b` как потребляющий их. Итак, для читаемости давайте реорганизуем этот фрагмент, разделив задачи на роли *производителя* и *потребителя*:

```js
// производитель:

var a = new LazyArray();

setInterval( function everySecond(){
    a.push( Math.random() );
}, 1000 );


// **************************
// потребитель:

var b = a.map( function double(v){
    return v * 2;
} );

b.listen( function onValue(v){
    console.log( v );
} );
```

`a` — это производитель, действующий по сути как поток значений. Мы можем рассматривать каждое значение, поступающее в `a`, как *событие*. Затем операция `map(..)` инициирует соответствующее событие в `b`, которое мы `listen(..)`, чтобы потреблять новое значение.

Причина, по которой мы разделяем задачи *производителя* и *потребителя*, — в том, что разные части нашего приложения могут отвечать за каждую из них. Такая организация кода может значительно улучшить как его читаемость, так и поддержку.

### Декларативное время

Мы очень осторожны в том, как вводим время в обсуждение. Конкретно, как промисы абстрагируют время от нашей заботы для единственной асинхронной операции, реактивное ФП абстрагирует (разделяет) время от серии значений/операций.

С перспективы `a` (производителя) единственная очевидная временнáя забота — это наш ручной цикл `setInterval(..)`. Но это только для демонстрационных целей.

Представьте, что `a` мог бы быть подключён к какому-то другому источнику событий: кликам мыши пользователя, нажатиям клавиш, WebSocket-сообщениям с сервера и т.д. В этом сценарии `a` не нужно беспокоиться о времени вообще. Он лишь независимый от времени канал для значений — когда бы они ни были готовы.

С перспективы `b` (потребителя) мы не знаем и не заботимся о том, когда и откуда поступают значения в `a`. По факту, все значения уже могут присутствовать. Нас интересует только то, что мы хотим эти значения — когда бы они ни были готовы. Снова — это независимая от времени (aka ленивая) модель операции `map(..)`.

Временнóе отношение между `a` и `b` является декларативным (и неявным!), а не императивным (или явным).

Ценность такой организации операций-во-времени может пока не ощущаться особенно эффективной. Давайте сравним с тем, как та же функциональность могла бы быть выражена императивно:

```js
// производитель:

var a = {
    onValue(v){
        b.onValue( v );
    }
};

setInterval( function everySecond(){
    a.onValue( Math.random() );
}, 1000 );


// **************************
// потребитель:

var b = {
    map(v){
        return v * 2;
    },
    onValue(v){
        v = this.map( v );
        console.log( v );
    }
};
```

Может казаться незначительным, но между этой более-императивной версией кода и предыдущей более-декларативной существует важное отличие — помимо того, что `b.onValue(..)` должен сам вызывать `this.map(..)`. В первом фрагменте `b` тянет из `a`, но во втором фрагменте `a` толкает в `b`. Иными словами, сравните `b = a.map(..)` с `b.onValue(v)`.

В последнем императивном фрагменте с точки зрения читаемости не ясно со стороны потребителя, откуда приходят значения `v`. Более того, жёсткое кодирование `b.onValue(..)` в середине логики производителя `a` является нарушением разделения ответственностей. Это может затруднить независимое рассуждение о производителе и потребителе.

Напротив, в первом фрагменте `b = a.map(..)` объявляет, что значения `b` берутся из `a`, трактуя `a` как абстрактный источник событий-данных, о котором нам не нужно беспокоиться в данный момент. Мы *декларируем*, что любое значение, поступающее из `a` в `b`, будет проходить через операцию `map(..)` как указано.

### Больше чем map

Для наглядности мы проиллюстрировали понятие сопряжения `a` и `b` во времени через однозначное отображение `map(..)`. Но многие другие наши ФП-операции также могут быть смоделированы во времени.

Рассмотрим:

```js
var b = a.filter( function isOdd(v) {
    return v % 2 == 1;
} );

b.listen( function onlyOdds(v){
    console.log( "Odd:", v );
} );
```

Здесь значение из `a` попадает в `b` только при прохождении предиката `isOdd(..)`.

Даже `reduce(..)` можно смоделировать во времени:

```js
var b = a.reduce( function sum(total,v){
    return total + v;
} );

b.listen( function runningTotal(v){
    console.log( "New current total:", v );
} );
```

Поскольку мы не задаём `initialValue` в вызове `reduce(..)`, ни редьюсер `sum(..)`, ни событийный колбэк `runningTotal(..)` не будут вызываться до тех пор, пока из `a` не поступит хотя бы два значения.

Этот фрагмент подразумевает, что свёртка имеет своего рода *память*: каждый раз, когда приходит следующее значение, редьюсер `sum(..)` будет вызываться с предыдущим `total` и новым значением `v`.

Другие ФП-операции, расширенные во времени, могут даже включать внутренний буфер: например, `unique(..)` отслеживает все встреченные до сих пор значения.

### Observable

Надеюсь, к этому моменту вы уже осознаёте важность реактивной, событийной, похожей на массив структуры данных, вроде вымышленного `LazyArray`, который мы придумали. Хорошая новость в том, что такая структура данных уже существует, и называется она **Observable**.

**Примечание:** Чтобы задать ожидания: следующее обсуждение — лишь краткое введение в мир Observable. Это значительно более глубокая тема, чем та, которую мы можем полностью рассмотреть здесь. Но если вы поняли Functional-Light Programming из этого текста и теперь уловили, как асинхронное время можно моделировать через принципы ФП, Observable должны последовать совершенно естественно в вашем дальнейшем обучении.

Observable реализованы в различных пользовательских библиотеках, наиболее известные из которых — [RxJS](https://github.com/Reactive-Extensions/RxJS) и [Most](https://github.com/cujojs/most). На момент написания существует находящееся в разработке предложение добавить Observable нативно в JS, точно так же как Promise были добавлены в ES6. Для демонстрации мы будем использовать Observable в стиле RxJS.

Вот наш ранний реактивный пример, выраженный через Observable вместо `LazyArray`:

```js
// производитель:

var a = new Rx.Subject();

setInterval( function everySecond(){
    a.next( Math.random() );
}, 1000 );


// **************************
// потребитель:

var b = a.map( function double(v){
    return v * 2;
} );

b.subscribe( function onValue(v){
    console.log( v );
} );
```

В мире RxJS наблюдатель (Observer) подписывается на наблюдаемое (Observable). Если объединить функциональность Observer и Observable, получится Subject. Поэтому, чтобы упростить фрагмент, мы конструируем `a` как Subject, чтобы иметь возможность вызывать на нём `next(..)` для помещения значений (событий) в его поток.

Если мы хотим держать Observer и Observable раздельно:

```js
// производитель:

var a = Rx.Observable.create( function onObserve(observer){
    setInterval( function everySecond(){
        observer.next( Math.random() );
    }, 1000 );
} );
```

В этом фрагменте `a` является observable, и, что неудивительно, отдельный observer называется `observer`; он способен «наблюдать» некоторые события (как наш цикл `setInterval(..)`); мы используем его метод `next(..)` для подачи событий в поток observable `a`.

Помимо `map(..)`, RxJS определяет более ста операторов, которые вызываются лениво при поступлении каждого нового значения. Как и с массивами, каждый оператор на Observable возвращает новый Observable, что означает возможность их объединения в цепочку. Если вызов функции-оператора определяет, что значение должно быть передано из входного Observable, оно будет испущено в выходной Observable; иначе оно отбрасывается.

Пример декларативной цепочки observable:

```js
var b =
    a
    .filter( v => v % 2 == 1 )      // только нечётные числа
    .distinctUntilChanged()         // только последовательно-различные
    .throttle( 100 )                // немного замедляем
    .map( v => v * 2 );             // удваиваем их

b.subscribe( function onValue(v){
    console.log( "Next:", v );
} );
```

**Примечание:** Присваивать observable переменной `b` и затем отдельно вызывать `b.subscribe(..)` вне цепочки не обязательно; это сделано здесь для того, чтобы подчеркнуть, что каждый оператор возвращает новый observable из предыдущего. Во многих примерах кода вызов `subscribe(..)` является просто последним методом в цепочке. Поскольку `subscribe(..)` технически мутирует внутреннее состояние observable, ФП-разработчики обычно предпочитают разделять эти два шага, чтобы более явно обозначить побочный эффект.

## Резюме

В этой книге подробно описано большое разнообразие ФП-операций, принимающих единственное значение (или немедленный список значений) и преобразующих их в другое значение/значения.

Для операций, которые будут выполняться во времени, все эти основополагающие ФП-принципы могут применяться независимо от времени. Точно так же как промисы моделируют единственное будущее значение, мы можем моделировать жадные списки значений как ленивые Observable-потоки (событийные потоки) значений, поступающих по одному.

`map(..)` на массиве запускает свою функцию отображения один раз для каждого значения, находящегося в массиве на данный момент, помещая все отображённые значения в итоговый массив. `map(..)` на Observable запускает свою функцию отображения один раз для каждого значения — когда бы оно ни поступило — и помещает все отображённые значения в выходной Observable.

Иными словами, если массив является жадной структурой данных для ФП-операций, то Observable — его ленивый-во-времени аналог.

**Примечание:** Для другого взгляда на асинхронное ФП, ознакомьтесь с библиотекой [**fasy**, которая обсуждается в Приложении C](apC.md/#bonus-2-fasy).
