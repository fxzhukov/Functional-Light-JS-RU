# Functional-Light JavaScript
# Глава 11: Всё вместе

К этому моменту у вас есть всё необходимое для понимания Functional-Light JavaScript. Новых концепций больше не будет.

В этой финальной главе наша главная цель — концептуальная целостность. Мы рассмотрим код, объединяющий многие из ключевых тем этой книги — применение того, что мы изучили. Прежде всего, этот пример кода призван проиллюстрировать подход «Functional Light» к JavaScript: баланс и прагматизм вместо догматизма.

Вам захочется практиковать эти техники самостоятельно — и как можно больше. Усвоение этой главы критически важно для применения принципов ФП в реальном коде.

## Подготовка

Давайте построим простой виджет биржевого тикера.

**Примечание:** Для справки: весь код этого примера находится в поддиректории `ch11-code/` — см. [репозиторий GitHub для этой книги](https://github.com/getify/Functional-Light-JS). Также выбранные ФП-хелперы, которые мы обсуждали на протяжении книги и которые нужны для этого примера, включены в `ch11-code/fp-helpers.js`. В этой главе мы сосредоточимся только на релевантных частях кода.

Для начала поговорим о разметке виджета — нам нужно место для отображения информации. Мы начинаем с пустого элемента `<ul ..>` в файле `ch11-code/index.html`, но во время работы DOM будет заполнен и выглядеть так:

```html
<ul id="stock-ticker">
    <li class="stock" data-stock-id="AAPL">
        <span class="stock-name">AAPL</span>
        <span class="stock-price">$121.95</span>
        <span class="stock-change">+0.01</span>
    </li>
    <li class="stock" data-stock-id="MSFT">
        <span class="stock-name">MSFT</span>
        <span class="stock-price">$65.78</span>
        <span class="stock-change">+1.51</span>
    </li>
    <li class="stock" data-stock-id="GOOG">
        <span class="stock-name">GOOG</span>
        <span class="stock-price">$821.31</span>
        <span class="stock-change">-8.84</span>
    </li>
</ul>
```

Прежде чем двигаться дальше, напомню: взаимодействие с DOM — это ввод-вывод (I/O), а значит — побочные эффекты. Мы не можем устранить эти побочные эффекты, но можем ограничивать и контролировать их. Мы будем очень намеренно минимизировать часть нашего приложения, работающую с DOM. Все эти техники мы изучали в [Главе 5](ch5.md).

Подводя итог функциональности виджета: код будет добавлять элементы `<li ..>` каждый раз при «получении» события о новой акции, и обновлять цену и изменение при поступлении событий обновления акции.

В примере кода из Главы 11, в файле `ch11-code/mock-server.js`, мы настраиваем несколько таймеров для отправки случайно сгенерированных фиктивных данных об акциях в простой эмиттер событий, имитируя получение сообщений о котировках с сервера. Мы предоставляем функцию `connectToServer()`, которая притворяется это делающей, но на самом деле просто возвращает экземпляр фиктивного эмиттера событий.

**Примечание:** Этот файл полностью фиктивный/модельный, поэтому я не тратил много усилий на его соответствие ФП. Я бы не рекомендовал тратить слишком много времени на код в этом файле. Если вы напишете настоящий сервер — очень интересное дополнительное упражнение для амбициозного читателя! — вы явно захотите уделить ему должное ФП-внимание.

В `ch11-code/stock-ticker-events.js` мы создаём некоторые Observable (через RxJS), подключённые к объекту эмиттера событий. Мы вызываем `connectToServer()`, чтобы получить этот эмиттер, затем слушаем события `"stock"` (добавление новой акции в тикер) и `"stock-update"` (обновление цены и изменения акции). Наконец, мы определяем преобразования входящих данных этих Observable, форматируя данные по мере необходимости.

В `ch11-code/stock-ticker.js` мы определяем поведение UI (побочный эффект DOM) как методы объекта `stockTickerUI`. Мы также определяем ряд хелперов: `getElemAttr(..)`, `stripPrefix(..)` и другие. Наконец, мы подписываемся (`subscribe(..)`) на два Observable, предоставляющих нам отформатированные данные для отрисовки в DOM.

## События акций

Давайте рассмотрим код в `ch11-code/stock-ticker-events.js`. Начнём с некоторых базовых хелперов:

```js
function addStockName(stock) {
    return setProp( "name", stock, stock.id );
}
function formatSign(val) {
    if (Number(val) > 0) {
        return `+${val}`;
    }
    return val;
}
function formatCurrency(val) {
    return `$${val}`;
}
```

Эти чистые функции должны быть достаточно просты для понимания. Вспомните, что [`setProp(..)` из Главы 4](ch4.md/#user-content-setprop) на самом деле клонирует объект перед установкой нового свойства. Это воплощает принцип из Главы 6: избегайте побочных эффектов, обращаясь со значениями как с неизменяемыми, даже если они таковыми не являются.

`addStockName(..)` используется для добавления свойства `name` в объект сообщения об акции, равного его `id`. Значение `name` впоследствии используется как видимое название акции в виджете.

Когда сообщение об акции получается с «сервера», оно выглядит примерно так:

```js
{ id: "AAPL", price: 121.7, change: 0.01 }
```

Перед отображением в DOM `price` нужно отформатировать через `formatCurrency(..)` (чтобы выглядело как `"$121.70"`), а `change` — через `formatChange(..)` (чтобы выглядело как `"+0.01"`). Но мы не хотим мутировать объект сообщения, поэтому нам нужен хелпер, который форматирует оба числа и возвращает новый объект акции:

```js
function formatStockNumbers(stock) {
    var stockDataUpdates = [
        [ "price", formatPrice( stock.price ) ],
        [ "change", formatChange( stock.change ) ]
    ];

    return reduce( function formatter(stock,[propName,val]){
        return setProp( propName, stock, val );
    } )
    ( stock )
    ( stockDataUpdates );
}
```

Мы создаём массив `stockDataUpdates` для хранения кортежей (просто массивов) с именем свойства и новым отформатированным значением — для `price` и `change` соответственно. Мы применяем `reduce(..)` (см. [Главу 9](ch9.md/#reduce)) по этому массиву, используя объект `stock` как `initialValue`. Мы деструктурируем кортеж в `propName` и `val`, затем возвращаем вызов `setProp(..)`, который возвращает новый клонированный объект с установленным свойством.

Теперь определим ещё несколько хелперов:

```js
var formatDecimal = unboundMethod( "toFixed" )( 2 );
var formatPrice = pipe( formatDecimal, formatCurrency );
var formatChange = pipe( formatDecimal, formatSign );
var processNewStock = pipe( addStockName, formatStockNumbers );
```

Функция `formatDecimal(..)` принимает число (например, `2.1`) и вызывает на нём метод `toFixed( 2 )`. Мы используем [`unboundMethod(..)` из Главы 9](ch9.md/#adapting-methods-to-standalones) для создания стандалон-метода с поздним связыванием.

`formatPrice(..)`, `formatChange(..)` и `processNewStock(..)` — всё это композиции через `pipe(..)`, каждая из которых компонует пару операций слева направо (см. [Главу 4](ch4.md)).

Для создания наших Observable (см. [Главу 10](ch10.md/#observables)) из нашего эмиттера событий нам понадобится хелпер — каррированный (см. [Главу 3](ch3.md/#one-at-a-time)) стандалон из `Rx.Observable.fromEvent(..)` библиотеки RxJS:

```js
var makeObservableFromEvent =
    curry( Rx.Observable.fromEvent, 2 )( server );
```

Эта функция настроена на прослушивание `server` (эмиттер событий) и ждёт строку с именем события, чтобы создать свой Observable. Теперь у нас все части на месте для создания наблюдателей для двух событий и отображения-трансформации этих наблюдателей для форматирования входящих данных:

```js
var observableMapperFns = [ processNewStock, formatStockNumbers ];

var stockEventNames = [ "stock", "stock-update" ];

var [ newStocks, stockUpdates ] = pipe(
    map( makeObservableFromEvent ),
    curry( zip )( observableMapperFns ),
    map( spreadArgs( mapObservable ) )
)
( stockEventNames );
```

Начинаем со `stockEventNames` — массива имён событий (`["stock","stock-update"]`), затем `map(..)` (см. [Главу 9](ch9.md/#map)) его в список двух Observable, `zip(..)` (см. [Главу 9](ch9.md/#zip)) с списком функций-маперов Observable, получая список кортежей вида `[ observable, mapperFn ]`. Наконец, `map(..)` эти кортежи через `mapObservable(..)`, разворачивая каждый кортеж как отдельные аргументы через `spreadArgs(..)` (см. [Главу 3](ch3.md/#user-content-spreadargs)).

Финальный результат — список из двух отображённых Observable, которые мы деструктурируем в присваивания для `newStocks` и `stockUpdates` соответственно.

Вот и всё — таков наш FP-Light-подход к настройке Observable событий тикера акций! Мы подпишемся на эти два Observable в `ch11-code/stock-ticker.js`.

## Пользовательский интерфейс тикера акций

Если вы чувствуете себя достаточно уверенно с ФП из предыдущего раздела, вы готовы погрузиться в `ch11-code/stock-ticker.js`. Этот файл значительно сложнее, поэтому мы не торопясь рассмотрим каждую его часть.

Начнём с определения некоторых хелперов, которые помогут нам с задачами DOM:

```js
function isTextNode(node) {
    return node && node.nodeType == 3;
}
function getElemAttr(prop,elem) {
    return elem.getAttribute( prop );
}
function setElemAttr(elem,prop,val) {
    // !!ПОБОЧНЫЕ ЭФФЕКТЫ!!
    return elem.setAttribute( prop, val );
}
function matchingStockId(id,node){
    return getStockId( node ) == id;
}
function isStockInfoChildElem(elem) {
    return /\bstock-/i.test( getClassName( elem ) );
}
function appendDOMChild(parentNode,childNode) {
    // !!ПОБОЧНЫЕ ЭФФЕКТЫ!!
    parentNode.appendChild( childNode );
    return parentNode;
}
function setDOMContent(elem,html) {
    // !!ПОБОЧНЫЕ ЭФФЕКТЫ!!
    elem.innerHTML = html;
    return elem;
}

var createElement = document.createElement.bind( document );

var getElemAttrByName = curry( getElemAttr, 2 );
var getStockId = getElemAttrByName( "data-stock-id" );
var getClassName = getElemAttrByName( "class" );
var isMatchingStock = curry( matchingStockId, 2 );
```

Это должно быть в основном понятно без пояснений.

Обратите внимание, что я выделил побочные эффекты мутации состояния DOM-элемента. Нам не так просто клонировать DOM-объект и заменить его, поэтому мы здесь соглашаемся на побочный эффект изменения существующего. По крайней мере, если у нас есть ошибка в рендеринге DOM, мы можем легко найти эти комментарии к коду, чтобы сузить круг подозреваемых.

Вот несколько других вспомогательных функций:

```js
function stripPrefix(prefixRegex,val) {
    return val.replace( prefixRegex, "" );
}

function listify(listOrItem) {
    if (!Array.isArray( listOrItem )) {
        return [ listOrItem ];
    }
    return listOrItem;
}
```

Давайте определим хелпер для получения дочерних узлов DOM-элемента:

```js
var getDOMChildren = pipe(
    listify,
    flatMap(
        pipe(
            curry( prop )( "childNodes" ),
            Array.from
        )
    )
);
```

Сначала мы используем `listify(..)` для обеспечения наличия списка элементов (даже если он состоит только из одного элемента). Вспомните [`flatMap(..)` из Главы 9](ch9.md/#user-content-flatmap), который отображает список и затем выравнивает список-списков в более плоский список.

Наша функция отображения здесь преобразует элемент в его список `childNodes`, который мы превращаем в настоящий массив (вместо живого NodeList) с помощью `Array.from(..)`. Эти две функции компонуются (через `pipe(..)`) в единую функцию-маппер — это слияние (fusion, см. [Главу 9](ch9.md/#fusion)).

Теперь используем этот хелпер `getDOMChildren(..)` для определения утилит извлечения конкретных DOM-элементов нашего виджета:

```js
function getStockElem(tickerElem,stockId) {
    return pipe(
        getDOMChildren,
        filterOut( isTextNode ),
        filterIn( isMatchingStock( stockId ) )
    )
    ( tickerElem );
}
function getStockInfoChildElems(stockElem) {
    return pipe(
        getDOMChildren,
        filterOut( isTextNode ),
        filterIn( isStockInfoChildElem )
    )
    ( stockElem );
}
```

`getStockElem(..)` начинает с DOM-элемента `tickerElem` нашего виджета, получает его дочерние элементы, затем фильтрует для получения элемента, соответствующего указанному идентификатору акции. `getStockInfoChildElems(..)` делает почти то же самое, только начинает с элемента акции и сужает с другими фильтрами.

Обе утилиты отфильтровывают текстовые узлы (поскольку они работают иначе, чем обычные DOM-узлы), и обе возвращают массив DOM-элементов, даже если это всего один элемент.

### Основной API

Мы будем использовать объект `stockTickerUI` для организации трёх основных методов манипуляции UI:

```js
var stockTickerUI = {

    updateStockElems(stockInfoChildElemList,data) {
        // ..
    },

    updateStock(tickerElem,data) {
        // ..
    },

    addStock(tickerElem,data) {
        // ..
    }
};
```

Сначала рассмотрим `updateStock(..)` — это самый простой из трёх:

```js
updateStock(tickerElem,data) {
    var getStockElemFromId = curry( getStockElem )( tickerElem );
    var stockInfoChildElemList = pipe(
        getStockElemFromId,
        getStockInfoChildElems
    )
    ( data.id );

    return stockTickerUI.updateStockElems(
        stockInfoChildElemList,
        data
    );
},
```

Каррирование хелпера `getStockElem(..)` с `tickerElem` даёт нам `getStockElemFromId(..)`, который будет получать `data.id`.

Через `pipe(..)` возвращаемое значение `getStockElemFromId(data.id)` является элементом `<li>` (фактически, списком, содержащим только этот элемент), который передаётся в `getStockInfoChildElems(..)`.

Результат — список (`stockInfoChildElemList`) с тремя дочерними элементами `<span>` для отображения информации об акции. Мы передаём этот список и объект `data` сообщения об акции в `stockTickerUI.updateStockElems(..)` для фактического обновления этих трёх элементов `<span>` новыми данными.

Теперь посмотрим, как определён `stockTickerUI.updateStockElems(..)`:

```js
updateStockElems(stockInfoChildElemList,data) {
    var getDataVal = curry( reverseArgs( prop ), 2 )( data );
    var extractInfoChildElemVal = pipe(
        getClassName,
        curry( stripPrefix )( /\bstock-/i ),
        getDataVal
    );
    var orderedDataVals =
        map( extractInfoChildElemVal )( stockInfoChildElemList );
    var elemsValsTuples =
        filterOut( function updateValueMissing([infoChildElem,val]){
            return val === undefined;
        } )
        ( zip( stockInfoChildElemList, orderedDataVals ) );

    // !!ПОБОЧНЫЕ ЭФФЕКТЫ!!
    compose( each, spreadArgs )
    ( setDOMContent )
    ( elemsValsTuples );
},
```

Согласен, много для восприятия. Но разберём по оператору.

`getDataVal(..)` привязан к объекту `data`, после каррирования с переставленными аргументами, поэтому теперь он ждёт имени свойства для извлечения из `data`.

Далее рассмотрим, как определён `extractInfoChildElemVal(..)`:

```js
var extractInfoChildElemVal = pipe(
    getClassName,
    curry( stripPrefix )( /\bstock-/i ),
    getDataVal
);
```

Эта функция принимает DOM-элемент, получает его CSS-класс, удаляет префикс `"stock-"` из этого значения, затем использует получившееся значение (`"name"`, `"price"` или `"change"`) как имя свойства для извлечения из объекта `data` через `getDataVal(..)`.

Это может казаться запутанным способом получить значения из объекта `data`. Но цель — иметь возможность извлекать эти значения из `data` в том же порядке, в котором элементы `<span>` появляются в списке `stockInfoChildElemList`; мы достигаем этого, используя `extractInfoChildElem(..)` как функцию отображения по этому списку DOM-элементов, называя результирующий список `orderedDataVals`.

Затем мы «застёгиваем» (`zip`) список `<span>`-ов обратно с упорядоченными значениями данных, создавая кортежи, в которых DOM-элемент и значение для его обновления связаны в пары:

```js
zip( stockInfoChildElemList, orderedDataVals )
```

Интересная деталь, которая совсем не была очевидна ранее: из-за того, как мы определили трансформации Observable, объекты сообщений о новых акциях будут иметь свойство `name` в `data`, соответствующее элементу `<span class="stock-name">`, но `name` будет отсутствовать в объектах сообщений об обновлении акций.

Если в объекте сообщения нет свойства, мы не должны обновлять соответствующий DOM-элемент. Поэтому нам нужно применить `filterOut(..)` для любых кортежей, где вторая позиция (значение данных) является `undefined`:

```js
var elemsValsTuples =
    filterOut( function updateValueMissing([infoChildElem,val]){
        return val === undefined;
    } )
    ( zip( stockInfoChildElemList, orderedDataVals ) );
```

Результат после этой фильтрации — список кортежей (вида `[ <span>, ".." ]`), готовых для обновления содержимого DOM, который мы присваиваем `elemsValsTuples`.

**Примечание:** Поскольку предикат `updateValueMissing(..)` указан здесь в виде inline-функции, мы контролируем его сигнатуру. Вместо использования `spreadArgs(..)` для адаптации его к развёртыванию одного аргумента-массива как двух отдельных именованных параметров, мы используем деструктуризацию массива параметров в объявлении функции (`function updateValueMissing([infoChildElem,val]){ ..`); см. [Главу 2](ch2.md/#user-content-funcparamdestr) для подробностей.

Наконец, нам нужно обновить содержимое DOM наших элементов `<span>`:

```js
// !!ПОБОЧНЫЕ ЭФФЕКТЫ!!
compose( each, spreadArgs )( setDOMContent )
( elemsValsTuples );
```

Мы итерируем список `elemsValsTuples` с помощью `each(..)` (см. [обсуждение `forEach(..)` в Главе 9](ch9.md/#mapping-vs-eaching)).

Вместо использования `pipe(..)` как в других местах, эта композиция использует `compose(..)` (см. [Главу 4](ch4.md/#user-content-generalcompose)) для передачи `setDomContent(..)` в `spreadArgs(..)`, а затем это передаётся как функция-итератор в `each(..)`. Каждый кортеж разворачивается как аргументы в `setDOMContent(..)`, которая соответствующим образом обновляет DOM-элемент.

Два из основных UI-методов рассмотрены, осталось один: `addStock(..)`. Определим его целиком, а затем рассмотрим по шагам, как делали ранее:

```js
addStock(tickerElem,data) {
    var [stockElem, ...infoChildElems] = map(
        createElement
    )
    ( [ "li", "span", "span", "span" ] );
    var attrValTuples = [
        [ ["class","stock"], ["data-stock-id",data.id] ],
        [ ["class","stock-name"] ],
        [ ["class","stock-price"] ],
        [ ["class","stock-change"] ]
    ];
    var elemsAttrsTuples =
        zip( [stockElem, ...infoChildElems], attrValTuples );

    // !!ПОБОЧНЫЕ ЭФФЕКТЫ!!
    each( function setElemAttrs([elem,attrValTupleList]){
        each(
            spreadArgs( partial( setElemAttr, elem ) )
        )
        ( attrValTupleList );
    } )
    ( elemsAttrsTuples );

    // !!ПОБОЧНЫЕ ЭФФЕКТЫ!!
    stockTickerUI.updateStockElems( infoChildElems, data );
    reduce( appendDOMChild )( stockElem )( infoChildElems );
    appendDOMChild( tickerElem, stockElem );
}
```

Этому UI-методу нужно создать базовую DOM-структуру для нового элемента акции, а затем использовать `stockTickerUI.updateStockElems(..)` для обновления его содержимого. Сначала:

```js
var [stockElem, ...infoChildElems] = map(
    createElement
)
( [ "li", "span", "span", "span" ] );
```

Мы создаём родительский `<li>` и три дочерних элемента `<span>`, присваивая их соответственно `stockElem` и списку `infoChildElems`.

Для инициализации этих элементов с соответствующими DOM-атрибутами создаём список списков-кортежей. Каждый элемент основного списка соответствует четырём DOM-элементам по порядку. Каждый подсписок содержит кортежи, представляющие пары атрибут-значение для установки на каждый соответствующий DOM-элемент:

```js
var attrValTuples = [
    [ ["class","stock"], ["data-stock-id",data.id] ],
    [ ["class","stock-name"] ],
    [ ["class","stock-price"] ],
    [ ["class","stock-change"] ]
];
```

Теперь хотим «застегнуть» (`zip(..)`) список из четырёх DOM-элементов с этим списком `attrValTuples`:

```js
var elemsAttrsTuples =
    zip( [stockElem, ...infoChildElems], attrValTuples );
```

Структура этого списка теперь выглядит примерно так:

```txt
[
    [ <li>, [ ["class","stock"], ["data-stock-id",data.id] ] ],
    [ <span>, [ ["class","stock-name"] ] ],
    ..
]
```

Если бы мы хотели императивно обрабатывать такую структуру данных для присваивания кортежей атрибут-значение каждому DOM-элементу, мы, вероятно, использовали бы вложенные циклы `for`. Наш ФП-подход будет похожим, но с вложенными итерациями `each(..)`:

```js
// !!ПОБОЧНЫЕ ЭФФЕКТЫ!!
each( function setElemAttrs([elem,attrValTupleList]){
    each(
        spreadArgs( partial( setElemAttr, elem ) )
    )
    ( attrValTupleList );
} )
( elemsAttrsTuples );
```

Внешний `each(..)` итерирует список кортежей, причём каждый `elem` и его ассоциированный `attrValTupleList` разворачиваются как именованные параметры в `setElemAttrs(..)` через деструктуризацию массива параметров, как объяснялось ранее.

Внутри этого внешнего цикла итерации подсписок кортежей атрибут-значение итерируется внутренним `each(..)`. Внутренняя функция-итератор — это развёртывание аргументов (каждого кортежа атрибут-значение) для частичного применения `setElemAttr(..)` с `elem` в качестве первого аргумента.

На этом этапе у нас есть список элементов `<span>` с атрибутами, но без содержимого `innerHTML`. Мы устанавливаем `data` в элементах `<span>` с помощью `stockTickerUI.updateStockElems(..)` — точно так же, как для события обновления акции.

Теперь нам нужно добавить эти `<span>`-ы к родительскому `<li>`, и мы делаем это через `reduce(..)` (см. [Главу 9](ch9.md/#reduce)):

```js
reduce( appendDOMChild )( stockElem )( infoChildElems );
```

Наконец, обычный старый побочный эффект DOM-мутации для добавления нового элемента акции в DOM виджета:

```js
appendDOMChild( tickerElem, stockElem );
```

### Подписка на Observable

Наша последняя главная задача — подписаться на Observable, определённые в `ch11-code/stock-ticker-events.js`, привязав эти подписки к соответствующим основным UI-методам (`addStock(..)` и `updateStock(..)`).

Во-первых, заметим, что каждый из этих методов ожидает `tickerElem` в качестве первого параметра. Давайте создадим список (`stockTickerUIMethodsWithDOMContext`), инкапсулирующий DOM-элемент тикера с каждым из этих двух методов через частичное применение (иначе говоря, [замыкание; см. Главу 2](ch2.md/#keeping-scope)):

```js
var ticker = document.getElementById( "stock-ticker" );

var stockTickerUIMethodsWithDOMContext = map(
    pipe( partialRight, unary )( partial, ticker )
)
( [ stockTickerUI.addStock, stockTickerUI.updateStock ] );
```

Сначала мы используем `partialRight(..)` (правостороннее частичное применение) на утилите `partial(..)`, предустанавливая её крайний правый аргумент как `ticker`. Затем эту правосторонне-частично-применённую функцию `partial(..)` передаём через `unary(..)`, чтобы защитить её от получения нежелательных лишних аргументов от `map(..)` (см. [Главу 3](ch3.md/#user-content-mapunary)), через `pipe(..)`. Результат — функция-маппер, ожидающая функцию для частичного применения (с одним аргументом: `ticker`). Эту функцию-маппер мы применяем через `map(..)` к функциям `stockTickerUI.addStock(..)` и `stockTickerUI.updateStock(..)` соответственно.

Результат `map(..)` — массив `stockTickerUIMethodsWithDOMContext`, содержащий две частично применённые функции; эти две функции теперь подходят в качестве обработчиков подписки на Observable.

Хотя мы используем замыкание для сохранения состояния `ticker` в этих двух функциях, в [Главе 7](ch7.md) мы видели, что могли бы «хранить» это значение `ticker` как свойство объекта, возможно, через привязку `this` каждой функции к `stockTickerUI`. Поскольку `this` является неявным входом (см. [Главу 2](ch2.md/#whats-this)) и обычно менее предпочтителен, я выбрал форму замыкания вместо объектной.

Для подписки на Observable создадим хелпер с несвязанным методом:

```js
var subscribeToObservable =
    pipe( unboundMethod, uncurry )( "subscribe" );
```

`unboundMethod("subscribe")` каррирован, поэтому мы применяем к нему `uncurry(..)` (см. [Главу 3](ch3.md/#no-curry-for-me-please)).

Теперь нам просто нужен список Observable, чтобы применить к нему `zip(..)` со списком DOM-привязанных UI-методов; получившиеся кортежи будут включать как Observable, так и функцию-слушатель для подписки. Мы обрабатываем каждый кортеж с `each(..)` и используем `spreadArgs(..)` (см. [Главу 3](ch3.md/#user-content-spreadargs)) для разворачивания содержимого кортежа как двух аргументов в `subscribeToObservable(..)`:

```js
var stockTickerObservables = [ newStocks, stockUpdates ];

// !!ПОБОЧНЫЕ ЭФФЕКТЫ!!
each( spreadArgs( subscribeToObservable ) )
( zip( stockTickerUIMethodsWithDOMContext, stockTickerObservables ) );
```

Технически мы мутируем состояние этих Observable, подписываясь на них, и, более того, используем `each(..)` — почти всегда ассоциированный с побочными эффектами! — поэтому мы отмечаем это комментарием к коду.
